## 1. 项目目标

`codex-switch` 是本机 Codex 运行态切换与多会话同步工具。

- 解决问题：在当前 Codex 账号登录态与一个 API 中转站调用态之间安全切换，并把本地历史会话同步成不绑定具体运行态的统一会话视图。
- 主要用户：当前 Windows 主机上的 Codex Desktop / Codex CLI 使用者。
- 成功产物：React + Tauri 桌面应用，具备 Codex 账号态保存、单个 API 中转站配置、关闭确认后真实替换 `auth.json` / `config.toml`、共享会话池热同步、备份和脱敏展示。
- 运行态输入：默认从 `CODEX_HOME` 解析 Codex home；未设置时使用 `%USERPROFILE%\.codex`。核心文件包括 `auth.json`、`config.toml`、`sessions\**\*.jsonl`、`session_index.jsonl`，以及位于 Codex `sqlite_home` 下的 `state_5.sqlite`、可选 `logs_2.sqlite` / `goals_1.sqlite`。`sqlite_home` 优先读取 `config.toml`，其次读取 `CODEX_SQLITE_HOME`，最后回落到 Codex home；`sqlite\codex-dev.db` 只作为兼容探测项，不作为当前会话同步来源。
- 运行态输出：工具自身加密运行态、备份快照、共享会话池和同步报告。

## 2. 运行模式

当前已实现 / 已接入的运行模式：

- 桌面正常模式：Tauri 启动 React UI，顶部在“运行态”和“会话管理”之间切换；用户在运行态页管理 Codex 账号态 / API 中转站态，在会话管理页查看、删除和恢复会话。
- Dry-run 模式：只扫描 `.codex` 与会话来源，不写入目标文件，输出同步计划和风险。
- 切换模式：检测 Codex 运行状态；如正在运行则弹窗确认关闭，关闭成功后先同步当前会话到共享会话池，再备份当前 `auth.json` / `config.toml` / `state_5.sqlite`，随后原子替换目标运行态的 `auth.json` 和完整 `config.toml`，最后把共享会话写回当前 Codex home。
- API 中转站模式：用户只配置一个中转站，保存 base URL、model 和 API key；未显式写 scheme 的 URL 默认补 `https://`。敏感 API key 在工具运行态中只进入加密 `auth.enc`，切换到 Codex 时写入 live `auth.json` 的 API key 登录态；live `config.toml` 只写 `openai_custom` provider 连接参数，不写 `api_key`、`env_key` 或 `goal`，UI 和文档不回显。
- 会话同步模式：单独点击“立即同步”时支持热同步，不要求关闭 Codex；后端先备份目标，再把当前 Codex home 与 `%APPDATA%\codex-switch\shared-sessions` 双向同步。同步采用 JSONL-first：先从 `sessions/**/*.jsonl` 的 `session_meta.payload.id` 建立有效会话集合，只同步存在正文 JSONL 的未归档会话；已归档会话只跳过同步，不自动删除，也不从 shared-sessions 清理旧副本。只有 SQLite 行但缺失正文 JSONL 的孤儿记录会计入跳过项，不写入目标。同步同时合并 `session_index.jsonl`，修复重复 thread 的缺失 JSONL / 错误 `rollout_path`，并按当前 `config.toml` 的 `model_provider` 归一化 `threads.model_provider` 与 JSONL `session_meta.payload.model_provider`。若 SQLite 被 Codex 独占锁住，则返回错误并可重试。
- 会话管理模式：默认展示“当前 Codex Home + shared-sessions”合并视图；同一会话 ID 同时存在时以当前 Codex Home 的标题、状态、时间和归档状态为准，shared-sessions 只补缺。来源列显示“本机 / 共享池 / 两边都有”。用户主动删除时后端先备份当前 Codex Home 与 shared-sessions，再从两边硬删除 SQLite `threads`、相关边表、JSONL 正文和 `session_index.jsonl`；已归档会话不额外二次确认，未归档会话必须在 UI 里二次确认。`恢复可见` 只更新当前 Codex Home 的 `threads.archived = 0` 和可选 `archived_at = NULL`，不立即强制同步。
- 恢复模式：当前已生成备份快照；会话管理页支持恢复归档可见，完整备份回滚 UI 仍是后续硬化项。

## 3. 配置加载链路

配置来源：

- Codex home 默认路径：优先 `CODEX_HOME`，否则 `%USERPROFILE%\.codex`，最后兜底 `HOME/.codex`；后续实现应允许高级设置覆盖。
- Codex SQLite home：优先读取 live `config.toml` 的 `sqlite_home`，其次读取环境变量 `CODEX_SQLITE_HOME`，最后回落到 Codex home。共享会话池内部固定使用自身目录，不继承用户侧 `sqlite_home`，避免共享池写到真实用户目录。
- 登录态：`auth.json`，含敏感 token，只能加密保存和脱敏展示。
- Codex 配置：`config.toml`，切换登录态或 API provider 时默认保护全局配置，只 patch 账号 / provider 绑定项。
- 工具配置：放在 `%APPDATA%\codex-switch`，保存 `runtimes\plus`、`runtimes\relay`、备份 manifest 和共享会话池。

敏感字段不得在日志、UI、文档或错误报告中明文输出。

## 4. 核心流程

MVP 核心流程：

1. 启动与预检：确认 Codex home、关键文件、SQLite schema、磁盘空间和 Codex 运行状态。
2. 已实现的只读扫描：Tauri command 调用 Rust 后端读取 `auth.json` 结构摘要、关键文件存在性、`state_5.sqlite.threads` 和 `sessions/**/*.jsonl`。
3. 已实现的 dry-run：按线程 ID 比较 source 与 target，输出新增/重复线程数，不写入真实 `.codex`。
4. 已实现的运行态存储：`runtime_store` 使用 Windows DPAPI 加密 Codex 账号态 / 中转站 `auth.json`，并保存每个运行态的完整 `config.toml`，占用 `%APPDATA%\codex-switch\runtimes\<plus|relay>`；`plus` 仅为内部兼容 ID，UI 显示为 Codex 账号态。
5. 已实现的备份：复制 `auth.json`、`config.toml`、`state_5.sqlite` 及可选 WAL/SHM，生成 `sessions-manifest.json` 和 SHA-256 清单。
6. 已实现的运行态切换：如 Codex 正在运行，前端弹窗确认后调用关闭；后端再次拒绝运行中切换，然后同步到共享会话池、快照当前状态、替换目标运行态的 `auth.json` / `config.toml`，再从共享会话池写回。
7. 已实现的会话写入同步：单独同步不关闭 Codex，后端先备份、再与共享会话池双向事务写入 SQLite、复制 JSONL 并合并 `session_index.jsonl`；同步以 JSONL 正文为可靠来源，跳过缺少 JSONL 正文的 DB-only 孤儿 thread。重复 thread 不再直接跳过，会继续修复目标侧缺失 JSONL 与错误 `rollout_path`。运行态切换流程内部仍会在关闭 Codex 后执行一次同步，并把当前 Codex home 的会话 provider 归一到目标运行态：Codex 账号态默认 `openai`，API 中转站态为 `openai_custom`。
8. 已实现的会话管理：前端会话管理页从后端读取当前 Codex Home 与 shared-sessions 的合并视图，删除动作热写入并先备份两边；归档会话安全硬删除，未归档会话必须二次确认；恢复可见只写当前 Codex Home 的归档字段，下一次同步再按未归档会话参与。
9. 后续恢复与审计：当前有备份产物；完整备份回滚 UI、审计日志、真实沙箱 E2E 仍需硬化。

## 5. 模块与边界

当前模块边界：

- `codex_paths`：统一解析 Codex home、`sqlite_home`、会话正文目录和 `session_index.jsonl`，封装用户真实路径与共享池本地路径的差异。
- `codex_home`：路径发现、文件探测、运行态检测。
- `runtime_store`：Codex 账号态 / 单个 API 中转站运行态、DPAPI 加密存储、完整 `config.toml` 保存。
- `crypto`：Windows DPAPI protect / unprotect。
- `process_control`：Codex 进程检测与关闭，不匹配 `codex-switch.exe`。
- `runtime_switcher`：共享会话池同步、`auth.json` / `config.toml` 原子替换和切换结果汇总；切换后按目标运行态 provider 调用 provider 感知会话同步，避免 Codex App 因 provider 过滤看不到历史会话。
- `profile_store` / `switcher`：旧 profile 后端兼容能力，当前主 UI 不再使用。
- `config_patch`：TOML 解析、保护字段、账号覆盖、diff。
- `session_scan`：按解析后的 `sqlite_home` 做 SQLite 只读扫描、dry-run 去重，并扫描 Codex home 下的 JSONL 正文。
- `session_sync`：JSONL-first 会话索引、JSONL 复制、`session_index.jsonl` 合并、SQLite 事务合并；从 `sessions/**/*.jsonl` 的 `session_meta.payload.id` 反查正文文件，跳过缺少正文的 DB-only 孤儿记录和已归档 thread，修复重复 thread 的缺失文件与错误路径，只改 `session_meta.payload.model_provider` 元数据，不全局替换用户/助手正文。
- `session_manager`：会话管理专用模块，合并当前 Codex Home 与 shared-sessions 视图，执行删除所选和恢复可见；删除前备份两边，删除范围覆盖 `threads`、相关边表、JSONL 正文和 `session_index.jsonl`，恢复只改当前 Codex Home。
- `backup`：按解析后的 `sqlite_home` 快照 `state_5.sqlite` 及 WAL/SHM，同时备份 `session_index.jsonl` 与 SHA-256 manifest。
- `redaction`：敏感字段脱敏。
- `commands`：Tauri command 薄边界，不承载业务逻辑。

## 6. 输出文件与状态流

工具自身输出目录：`%APPDATA%\codex-switch`。

- `runtimes/plus/runtime.json`：Codex 账号态元数据，不含敏感明文；目录名 `plus` 是内部兼容 ID。
- `runtimes/plus/auth.enc`：加密后的 Codex 账号态 `auth.json`。
- `runtimes/plus/config.toml`：Codex 账号态完整配置。
- `runtimes/relay/runtime.json`：单个 API 中转站元数据，不含 API Key。
- `runtimes/relay/auth.enc`：加密后的中转站 API Key 登录态。
- `runtimes/relay/config.toml`：中转站运行态配置模板，包含 `model_provider = "openai_custom"` 与 `[model_providers.openai_custom]` 连接参数，但不持久化明文 API Key，也不写 `env_key`。当前 Codex CLI strict-config 实测不接受 provider 内 `api_key` 和 `goal` 字段；同时实测只要 live `auth.json` 处于 API key 登录态，config 不写 `env_key` 也可直接调用。
- `shared-sessions/state_5.sqlite`、`shared-sessions/sessions/` 与 `shared-sessions/session_index.jsonl`：Codex 账号态与中转站共用的会话池。
- `backups/<timestamp>-<operation-id>/`：切换、同步、删除会话或恢复可见前快照。
- `logs/audit.jsonl`：脱敏审计日志，后续硬化项。

## 7. 常见故障归因

- 登录态切换后不生效：先查 Codex 是否仍在运行、`auth.json` 修改时间、`config.toml` 是否被运行态覆盖。
- 会话列表缺失：先查活跃库是否真在 `sqlite_home/state_5.sqlite`，再查 `threads.rollout_path` 指向的 JSONL 是否存在、`session_index.jsonl` 是否包含对应会话，最后查 `threads.model_provider` 是否与当前 `config.toml` 的 `model_provider` 一致；Codex App 会按 provider / service 过滤，会话正文 JSONL 中也需要同步 `session_meta.payload.model_provider`。
- 归档会话没有同步回来：这是预期行为；归档会话只在会话管理页可见并可主动删除或恢复可见，不会被自动同步或清理。
- 同步重复：查共享会话池、`threads.id`、JSONL `session_meta.payload.id` 和文件哈希。
- SQLite 写入失败：查 `sqlite_home` 解析结果、schema fingerprint、锁状态、备份快照和事务错误。
- 配置损坏：查 TOML 解析错误和最近一次 config backup。

## 8. 测试与验证

当前真实可用验证命令：

- `npm test -- --run`：Vitest 前端 dashboard 测试。
- `npm run typecheck`：TypeScript 静态检查。
- `npm run build`：Vite 前端生产构建。
- `cargo test --manifest-path src-tauri/Cargo.toml`：Rust 后端单元测试。
- `npm run tauri -- build`：完整 Tauri 桌面应用 release 编译，当前输出 `src-tauri\target\release\codex-switch.exe`。

后续硬化仍需补充：

- 临时 Codex home 沙箱端到端测试：已用真实 CLI 验证“账号态可调用 → 中转站态可调用 → 切回账号态可调用”的基本路径；后续应把该流程沉淀为可重复自动化脚本，且不直接拿真实 `.codex` 做首轮写入。
- 真实沙箱端到端幂等性测试：重复同步不产生重复线程。
- 切换失败回滚测试。
- 敏感信息扫描：日志和 UI 输出不得含 token/API key。
- 中文乱码检查：所有中文文档、UI 文案、日志提示无可见乱码。
