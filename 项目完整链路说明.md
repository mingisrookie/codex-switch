## 1. 项目目标

`codex-switch` 是本机 Codex 运行态切换与多会话同步工具。

- 解决问题：在当前 Codex 账号登录态与一个 API 中转站调用态之间安全切换，把本地历史会话同步成统一视图，并为用户的 Codex 安装可配置的 Image2/Grok 能力。
- 主要用户：当前 Windows 主机上的 Codex Desktop / Codex CLI 使用者。
- 成功产物：React + Tauri 桌面应用，具备固定双槽位、真实运行态判定、关闭确认后的原子切换、共享会话池热同步、会话管理、固定 Skill 安装/配置、DPAPI 凭据、失败补偿、typed receipt 和脱敏操作记录。
- 运行态输入：默认从 `CODEX_HOME` 解析 Codex home；未设置时使用 `%USERPROFILE%\.codex`。核心文件包括 `auth.json`、`config.toml`、`sessions\**\*.jsonl`、`session_index.jsonl`，以及位于 Codex `sqlite_home` 下的 `state_5.sqlite`、可选 `logs_2.sqlite` / `goals_1.sqlite`。`sqlite_home` 优先读取 `config.toml`，其次读取 `CODEX_SQLITE_HOME`，最后回落到 Codex home；`sqlite\codex-dev.db` 只作为兼容探测项，不作为当前会话同步来源。
- 运行态输出：工具自身加密运行态、加密备份快照、共享会话池、操作记录以及面向 UI 的 dry-run/typed receipt。

## 2. 运行模式

当前已实现 / 已接入的运行模式：

- 桌面正常模式：Tauri 启动 React UI，顶部在“运行态 / 会话管理 / 技能”之间切换。Dashboard 的七个数据域保持独立 `loading | ready | error`；技能页仅首次进入时单独加载，技能错误不污染 Dashboard，Dashboard 错误也不在技能页展示。
- 固定双槽位模式：只管理 `plus`（Codex 账号，内部兼容 ID）和 `relay`（API 中转站）两个槽位。账号导入必须验证 live `auth_mode = chatgpt`；覆盖必须确认并归档旧的加密槽位文件。UI 分开显示已保存、当前精确匹配/模式匹配和最近验证。
- Dry-run 模式：只比较 current ↔ shared 双向未归档会话，返回新增与重复数量，不写入任何 Codex 文件；用户确认后才进入同步 mutation。
- 切换模式：前后端检查 Codex 进程；relay 目标先验证 `/models`，后端在真正写入前再次检查。随后为 current/shared 创建已验证快照，把 current 会话写入 shared，原子替换 `auth.json` 并在 live `config.toml` 上应用 runtime overlay，再从 shared 写回 current，最后要求目标运行态 `confidence = exact`。任何一步失败都恢复两份快照并报告 `rolledBack` / rollback failed。只有目标已经 exact 时才 no-op；mode-only 必须重新应用。
- API 中转站模式：用户只配置一个中转站，保存 base URL、model 和 API key；URL 必须是 HTTP(S)，不得内嵌凭据、query 或 fragment，未写 scheme 时补 `https://`。首次保存必须有 Key，后续空 Key 保留已加密值；UI 不回填。配置使用原生 `<dialog>` 和可访问名称/说明；保存失败保留本次输入便于重试，保存成功或取消关闭后销毁对话框内 Key。连接验证用 rustls 客户端携带 Bearer 请求 `<base>/models`，10 秒超时且禁止重定向，错误不回显 Key 或响应正文。
- 会话同步模式：单独点击“立即同步”允许 Codex 正在运行；后端先为 current/shared 创建完整快照，再双向同步。同步采用 JSONL-first：只同步存在有效正文 JSONL 的未归档会话，跳过 DB-only 孤儿和归档会话；SQLite 目标使用 2 秒 busy timeout、事务和 `PRAGMA quick_check`。热同步不会为 provider 变化或增长内容重写已有 live JSONL；关闭态、允许替换时，目标是源的严格旧前缀可原子替换为增长版本。内容分叉时按内容哈希保留独立文件。`session_index.jsonl` 只追加 allowlist 中的缺失 ID，并保证换行边界正确。热同步失败时只恢复 shared-sessions，live current Home 保持不动以保护并发变化，并在错误中给出 current 安全备份。
- 会话管理模式：默认展示 current + shared 合并视图，current 的显示元数据优先；支持搜索、排序、每页 50 条和跨页选择。所有硬删除都必须确认并要求 Codex 已关闭；在两根完整备份后，用 Immediate 事务删除 `threads` 与相关边表并原子更新 JSONL/index，跨根部分失败恢复两份快照。`恢复可见` 也要求 Codex 已关闭，只更新 current 的 `archived = 0` / `archived_at = NULL`，shared-only 选择不执行无效恢复。
- Skill 模式：固定管理 `image2` 与 `grokSearch` 两个 wire ID，分别落到 `CODEX_HOME\skills\newapi-image2-client` 与 `CODEX_HOME\skills\grok-search`。Image2 来源 ZIP 的审查 SHA-256 固定为 `648C192C2414BBFD9DBA36E264C01932BDCF7E2057A8BA2DA7006B40A94B332B`；Grok 包移除本机路径、endpoint 和配置。包体通过 `include_bytes!` 进入 EXE，不在运行时下载或复制管理员目录。安装/更新要求 Codex 已关闭和 mutation guard；本地漂移/未知目录需确认，旧目录先移动到同卷备份再原子激活 stage；原子 journal 让下次安装能处理进程中断留下的 stage/backup/live 组合。URL/Key 配置由用户输入，Key 仅保存 DPAPI 密文；首次必填、后续空值保留。
- 更新检查模式：每次应用启动只在当前进程内发起一次非阻塞检查，手动入口可再次触发；不注册后台服务、不在运行中轮询。Rust 后端固定请求本仓库最新正式 GitHub Release，8 秒超时、禁止重定向、限制 256 KiB、拒绝 draft/prerelease/非法 SemVer；前端显示当前/最新版本和可关闭横幅，下载动作只打开固定 Releases 页面，不自动下载或替换 EXE。
- 完整恢复模式：Dashboard 扫描可读 manifest 元数据，按 `createdAtMs` 排序后只对最近 5 个候选做 payload 大小/SHA-256 强校验，无效候选跳过，避免对全部历史 payload 线性哈希；UI 展示通过校验的候选，并明示“仅校验并展示最近 5 份备份候选；旧备份不会自动清理。”真正恢复时后端再次完整校验，并只接受受管 backup root 的直接子目录，按 `sourceRoot` 恢复 current 或 shared；恢复前创建安全快照，失败时用安全快照补偿。恢复按备份内 config 解析 `sqlite_home`、移除快照外文件，并执行 SQLite `quick_check`。当前没有自动 retention/prune，界面展示最近备份不代表旧目录被删除，受管备份会累积。

## 3. 配置加载链路

配置来源：

- Codex home 默认路径：优先 `CODEX_HOME`，否则 `%USERPROFILE%\.codex`，最后兜底 `HOME/.codex`；后续实现应允许高级设置覆盖。
- Codex SQLite home：优先读取 live `config.toml` 的 `sqlite_home`，其次读取环境变量 `CODEX_SQLITE_HOME`，最后回落到 Codex home。共享会话池内部固定使用自身目录，不继承用户侧 `sqlite_home`，避免共享池写到真实用户目录。
- 登录态：`auth.json`，含敏感 token，只能加密保存和脱敏展示。
- Codex 配置：`config.toml`，切换登录态或 API provider 时默认保护全局配置，只 patch 账号 / provider 绑定项。
- 工具配置：放在 `%APPDATA%\codex-switch`，保存运行态、备份、共享会话池、Skill 非敏感 config、Skill DPAPI credential 和 `logs\operations.jsonl`。

敏感字段不得在日志、UI、文档或错误报告中明文输出。

## 4. 核心流程

所有 mutation 使用同一语义：`preflight -> plan/dry-run -> backup -> apply -> verify -> receipt`；关闭态跨文件/跨根 mutation 发生失败时进入 `rollback -> verify rollback -> terminal receipt/error`。热同步是例外：恢复内部 shared 根但不把旧快照写回正在变化的 live current Home。恢复可见是单条 SQLite 更新，写前仍备份，SQL 自身保持原子性。

1. 启动与扫描：`loadDashboard` 并发调用七个 Dashboard 加载 command（含最近操作记录），并用 `Promise.allSettled` 保留各域真实错误；独立启动 effect 同时读取应用版本并异步检查更新，更新失败不进入 Dashboard 七域也不阻塞首屏。每个写入口仅按自身依赖域启用，恢复入口不依赖损坏的 Codex Home 扫描。备份列表加载可能顺带迁移 legacy plaintext auth payload，因此不把整组加载命令泛称为纯只读；迁移在 mutation guard 临界区内完成，释放锁后才扫描 manifest 并强校验最近 5 个候选，避免把昂贵哈希纳入写锁。所有 mutation command 先获取进程内 try-lock，再持有 Windows 独占 `%APPDATA%\codex-switch\mutation.lock` 文件句柄；同进程或第二个进程已有 mutation 时立即拒绝。RAII 句柄在正常结束或进程崩溃后由 Windows 自动释放，不依赖删除 lock 文件。
2. 计划：运行态切换解析 auth/overlay；会话同步输出 dry-run；完整恢复校验受管来源；Skill 安装只根据固定 ID 选择编译期包和后端推导的固定目标，配置命令在后端归一化 URL 并判断首次 Key。
3. 备份：`backup` 生成 manifest v2。`auth.json`、`config.toml`、SQLite online snapshot、`session_index.jsonl` 和所有会话 JSONL payload 均用当前 Windows 用户 DPAPI 加密；manifest 记录 source、相对路径、大小和 SHA-256。共享池显式使用 local paths，不继承用户 `sqlite_home`。
4. 应用：文件修改统一通过同目录临时文件、flush/sync 和原子替换；Windows 使用 `MoveFileExW` write-through。Skill 包先写同卷 stage，逐文件验 hash；覆盖时把旧目录移动到当前 Codex Home 的 `.codex-switch\skill-backups`，再 rename stage。Skill Key 经 DPAPI 后 Base64 写入工具配置目录。
5. 验证：备份/恢复检查大小和 SHA-256；同步/恢复执行 `PRAGMA quick_check`；运行态切换要求 exact；Skill 安装重新扫描 package manifest、固定 allowlist 和文件 SHA-256；高风险 mutation 再检查 Codex 已关闭。
6. 回执与操作记录：成功结果返回 typed receipt（operation ID、备份、计数、rolledBack、warnings）；失败返回脱敏错误，回滚/回滚失败终态写入本地 `operations.jsonl`。操作记录只含枚举 action/phase/status、时间、备份目录和计数；Dashboard 独立加载最近 20 条并展示最近 10 条的动作、终态、操作 ID、完成时间和备份路径。命令结束后立即释放 UI busy，Dashboard 刷新在后台 best-effort 执行；操作成功但刷新失败时保留成功回执并单独提示，操作失败时保留原错误并仅后台更新持久化历史。

## 5. 模块与边界

当前模块边界：

- `codex_paths`：统一解析 Codex home、`sqlite_home`、会话正文目录和 `session_index.jsonl`，封装用户真实路径与共享池本地路径的差异。
- `codex_home`：路径发现、文件探测、运行态检测。
- `runtime_store`：固定账号/relay 双槽位、DPAPI 认证存储、覆盖历史、验证/使用时间和最小 runtime overlay；不保存无关 live 全局配置。
- `crypto`：Windows DPAPI protect / unprotect。
- `process_control`：Codex 进程检测与关闭，不匹配 `codex-switch.exe`。
- `runtime_switcher`：双根快照、共享会话池同步、auth/config overlay 原子应用、exact 后置校验和双根失败补偿。
- `profile_store` / `switcher`：旧 profile 后端兼容能力，当前主 UI 不再使用。
- `config_patch`：TOML 解析、保护字段、账号覆盖、diff。
- `session_scan`：按解析后的 `sqlite_home` 做 SQLite 只读扫描、dry-run 去重，并扫描 Codex home 下的 JSONL 正文。
- `session_sync`：JSONL-first allowlist、SQLite 事务/quick_check、热写与关闭态写入差异、增长/分叉 JSONL 处理、append-only session index 和 provider 元数据流式原子重写。
- `session_manager`：current/shared 合并视图、所有硬删除确认、Immediate 事务、文件原子更新、双根补偿和 current-only 可见恢复。
- `backup`：manifest v2、DPAPI 全量 payload、SQLite Online Backup、校验/列表/精确恢复、安全清理和 legacy plaintext auth 迁移。
- `file_ops`：同目录临时文件、同步和平台原子替换的唯一实现。
- `relay_verify`：relay `/models` 脱敏连通性验证。
- `operation_log`：结构化终态操作记录。
- `skill_manager`：固定 Skill 包、manifest/hash/漂移状态、路径/reparse 校验、同卷 stage/备份/原子激活、URL 归一化和 DPAPI 配置存储。
- `update_check`：固定 GitHub latest-release 读取、stable SemVer 验证、响应大小/超时/重定向边界、并发检查 guard、限长说明和固定下载页。
- `redaction`：敏感字段脱敏。
- `commands`：Tauri command 薄边界，负责固定路径/槽位、进程门禁、恢复选择限制、typed receipt 组装和终态记录；不允许前端传任意 Codex 路径。

## 6. 输出文件与状态流

工具自身输出目录：`%APPDATA%\codex-switch`。

- `runtimes/plus/runtime.json`：Codex 账号态元数据，不含敏感明文；目录名 `plus` 是内部兼容 ID。
- `runtimes/plus/auth.enc`：加密后的 Codex 账号态 `auth.json`。
- `runtimes/plus/config.toml`：账号态 runtime overlay，只含切换需要的模型/provider 绑定。
- `runtimes/relay/runtime.json`：单个 API 中转站元数据，不含 API Key。
- `runtimes/relay/auth.enc`：加密后的中转站 API Key 登录态。
- `runtimes/relay/config.toml`：中转站 runtime overlay，包含 `model_provider = "openai_custom"` 与 provider 连接参数，不含无关 live 全局配置、明文 API Key、`env_key` 或 `goal`。
- `shared-sessions/state_5.sqlite`、`shared-sessions/sessions/` 与 `shared-sessions/session_index.jsonl`：Codex 账号态与中转站共用的会话池。
- `backups/<timestamp>-<pid>-<counter>-<reason>/manifest.json`：v2 恢复元数据、来源、大小与 SHA-256；同目录 `.enc` 文件是 DPAPI 加密 payload。
- `logs/operations.jsonl`：追加式脱敏操作记录；不保存凭据、请求/响应正文或自由文本错误。
- `skills/image2/config.json`、`skills/grok-search/config.json`：位于 `%APPDATA%\codex-switch` 下的非敏感 URL、固定模型和 credential 路径；不含 Key。
- `skills/<skill>/credential.enc`：当前 Windows 用户的 DPAPI 密文，经 Base64 供安装后的 PowerShell helper 解密；UI 只返回配置布尔状态。
- 当前 `CODEX_HOME\skills\newapi-image2-client` / `grok-search`：受管 Skill 文件和 `.codex-switch-package.json`；不含用户配置或 Key。
- 当前 `CODEX_HOME\.codex-switch\skill-backups\<operation-id>`：覆盖前的完整旧 Skill 目录；当前不自动清理。
- 当前 `CODEX_HOME\.codex-switch\skill-transactions\<skill>.json`：目录 swap 的原子恢复 journal；成功后删除，异常残留由下一次该 Skill 安装先恢复。

## 7. 常见故障归因

- 登录态切换后不生效：先查 Codex 是否仍在运行、relay `/models` 验证、操作 ID/终态、`auth.json` 修改时间，以及运行态检测是 exact、mode 还是 unknown；mode-only 需要重新应用。
- 会话列表缺失：先查活跃库是否真在 `sqlite_home/state_5.sqlite`，再查 `threads.rollout_path` 指向的 JSONL 是否存在、`session_index.jsonl` 是否包含对应会话，最后查 `threads.model_provider` 是否与当前 `config.toml` 的 `model_provider` 一致；Codex App 会按 provider / service 过滤，会话正文 JSONL 中也需要同步 `session_meta.payload.model_provider`。
- 归档会话没有同步回来：这是预期行为；归档会话只在会话管理页可见并可主动删除或恢复可见，不会被自动同步或清理。
- 同步重复：查共享会话池、`threads.id`、JSONL `session_meta.payload.id` 和文件哈希。
- SQLite 写入失败：查 `sqlite_home` 严格解析结果、schema、2 秒 busy timeout、事务/quick_check、操作回执和两根备份。
- 配置损坏：无效 live TOML 会阻断路径解析/写操作；查解析错误、操作记录和最近已验证备份，不允许静默回落到默认数据库。
- 操作成功但页面提示错误：若回执已出现且文案为“状态刷新失败”，mutation 已成功，失败的是后台 Dashboard 刷新；操作按钮不会继续被刷新占用。按操作 ID 和备份记录排查，不要重复执行。
- Skill 已安装但不可用：先检查页面是否显示“API Key 已加密配置”，再确认重启过 Codex；Image2 URL 必须以 `/v1` 结尾，Grok URL 在保存时会移除末尾 `/v1` 并由脚本调用 `/v1/responses`。不要读取 credential 文件排查明文。
- Skill 显示“本地已修改 / 非受管目录 / 安装异常”：工具不会静默覆盖；确认目录确实可被接管后使用覆盖安装，并保留回执中的旧目录备份。

## 8. 测试与验证

当前真实可用验证命令：

- `npm test -- --run`：Vitest 前端 Dashboard、会话与技能页测试。
- `npm run typecheck`：TypeScript 静态检查。
- `npm run build`：Vite 前端生产构建。
- `cargo fmt --manifest-path src-tauri/Cargo.toml -- --check`：Rust 格式门禁。
- `cargo clippy --manifest-path src-tauri/Cargo.toml --all-targets -- -D warnings`：Rust lint 门禁。
- `cargo test --manifest-path src-tauri/Cargo.toml`：Rust 后端单元测试。
- `cargo test --manifest-path src-tauri/Cargo.toml --test skill_manager_contract`：临时 `CODEX_HOME` Skill 安装、漂移、备份、URL、DPAPI 与 Windows PowerShell 兼容合同。
- `npm run tauri -- build`：完整 Tauri 桌面应用 release 编译，当前输出 `src-tauri\target\release\codex-switch.exe`。

`.github/workflows/ci.yml` 在 `windows-latest` / Node 22 / stable Rust 上自动执行前端测试、类型检查、构建以及 Rust fmt、clippy 和测试。临时 `CODEX_HOME` E2E、故障注入、恢复演练、敏感扫描和中文乱码检查仍应在发布回执中单独给证据，不能用“CI 文件存在”代替实际通过。

## 9. Release 发布链路

当前公开 release 采用 GitHub Releases + Windows 单文件可执行产物：

1. 同步版本号：`package.json`、`package-lock.json`、`src-tauri/Cargo.toml`、`src-tauri/Cargo.lock`、`src-tauri/tauri.conf.json`。
2. 同步公开文档：`README.md` release badge / 功能说明、`CHANGELOG.md`、根目录长期文档。
3. 运行完整验证：`npm test -- --run`、`npm run typecheck`、`npm run build`、Rust fmt/clippy/test、必要的临时 `CODEX_HOME` E2E/故障注入/恢复演练、敏感扫描、乱码检查和 `npm run tauri -- build`。
4. 提交并推送版本提交。
5. 创建版本 tag，例如 `v0.1.3`，并推送 tag。
6. 用 `gh release create` 或 `gh release edit` 发布，上传 `src-tauri\target\release\codex-switch.exe`。
7. 用 `gh release view <tag>` 和 `git ls-remote --tags origin <tag>` 验证远端 release/tag 真实存在。

应用内更新检查依赖该链路的正式 release/tag 版本一致性。不得只推 tag 不建 Release，也不得让 Release tag、Rust crate、Tauri config 和 npm 版本漂移。

## 10. 发布前剩余门禁

- 必须从干净工作区实际运行本节列出的前端/Rust门禁；不能只依赖历史结果或 CI 配置。
- 临时 `CODEX_HOME` E2E 至少覆盖账号态 → relay → 账号态、双向同步幂等和完整备份恢复；不得用真实 `.codex` 作为首轮写入目标。
- 故障注入至少覆盖运行态文件写入后失败、跨 current/shared 删除部分失败和恢复失败的安全快照补偿。
- 敏感扫描必须覆盖 UI 构建产物、操作记录、备份 manifest 和错误文本；不得出现 token/API Key/响应正文。
- 中文乱码检查必须覆盖本轮修改的 Markdown、TSX 和 Rust 文案。
- 已知产品残余：尚无 backup retention/prune、容量上限或安全清理入口；发布说明不得暗示旧备份会自动删除。

## DXM 大开发工作流链路

中大型开发、新模块、跨多文件重构、需求不清楚的任务，默认走：

1. 先用 `project-grill` 问清楚；有代码/文档时用 `grill-with-docs`，空项目用 `new-project-grill`，小脚本/demo 用 `lightweight-grill`。

4. 用 `.trellis/scripts/task.py start <task>` 进入 Trellis active task。
5. 按 Trellis 的 implement/check/update-spec/finish 节奏开发、验证、沉淀规范。
6. 任务完成后同步 DXM 长期文档；不能只更新 `.trellis/` 内部状态。
